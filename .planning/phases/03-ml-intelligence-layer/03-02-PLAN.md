<task type="auto">
  <name>Task 1: HMM regime model with fit and predict</name>
  <files>src/ml/regime.py, tests/test_regime.py, requirements.txt</files>
  <action>
Add `hmmlearn>=0.3.0` to requirements.txt. Install it.

Create `src/ml/regime.py` with a `RegimeDetector` class:

1. `__init__(self, n_states: int = 2, random_state: int = 42)` — creates a GaussianHMM with `n_states` states, full covariance type, and fixed random seed for reproducibility.

2. `fit(self, returns: np.ndarray, vix: np.ndarray) -> None` — fits the HMM on a 2D feature matrix [daily_returns, vix_level]. Standardize features before fitting (store scaler params as attributes for later transform). Sort states by volatility (state 0 = lowest vol = risk-on) so state labels are consistent across refits. Require minimum 252 observations (1 year). Store the fitted model as `self.model`.

3. `predict(self, returns: np.ndarray, vix: np.ndarray) -> dict` — given recent observations, returns:
   - `state`: int (0=risk-on, 1=risk-off, 2=crisis if 3-state)
   - `state_name`: str ("risk-on", "risk-off", "crisis")
   - `probabilities`: list[float] — filtered probability of each state
   - `transition_matrix`: list[list[float]] — transition probabilities
   - `expected_duration`: dict[str, float] — expected days in each regime (1 / (1 - P(stay)))

4. `select_n_states(self, returns: np.ndarray, vix: np.ndarray, candidates: list[int] = [2, 3]) -> int` — fit models for each candidate n_states, return the one with lowest BIC. Use this to decide 2 vs 3 states.

5. `save(self, path: str) -> None` / `load(self, path: str) -> None` — serialize/deserialize the fitted model + scaler params using pickle. Save to `config.ml_features_dir/regime_model.pkl`.

State naming convention after sorting by volatility:
- State 0: "risk-on" (lowest vol)
- State 1: "risk-off" (highest vol for 2-state, medium for 3-state)
- State 2: "crisis" (highest vol, only for 3-state)

Tests in `tests/test_regime.py`:
- Fit on synthetic data with clear regime structure (low-vol block + high-vol block)
- Verify state sorting (state 0 always has lower variance)
- Verify predict returns correct dict structure
- Verify probabilities sum to 1.0
- Verify save/load roundtrip preserves predictions
- Verify select_n_states returns valid candidate
- Edge case: insufficient data raises ValueError

<task type="auto">
  <name>Task 2: Regime update pipeline and feature store integration</name>
  <files>src/ml/regime.py, src/ml/feature_store.py, tests/test_regime.py</files>
  <action>
Add a `RegimeManager` class to `src/ml/regime.py` that orchestrates regime detection as a daily pipeline:

1. `__init__(self, feature_store: FeatureStore, model_dir: str)` — takes feature store for persistence and model directory for artifacts.

2. `async initialize(self, historical_returns: list[float], historical_vix: list[float]) -> None` — initial model fitting. Use select_n_states to pick 2 vs 3. Fit the model, save to disk. This should be called once when the system first starts or when retraining is needed.

3. `async update(self, latest_return: float, latest_vix: float, ticker: str = "SPX") -> dict` — daily update: append new observation to history, re-run predict with full history, save regime state + probability to feature store via `save_features()` (update `regime_state` and `regime_probability` columns), return prediction dict. This method is what the scheduler calls daily.

4. `async get_current_regime(self, ticker: str = "SPX") -> dict | None` — read latest regime state from feature store. Returns None if no regime data available yet.

5. `async should_retrain(self, ticker: str = "SPX") -> bool` — returns True if model hasn't been retrained in 30+ days or if regime transition rate is abnormally high (structural break indicator). Use a simple heuristic: if >3 regime transitions in last 5 days, suggest retraining.

The RegimeManager keeps a rolling window of returns + VIX in memory (last 1000 observations) so predict() doesn't need to query the database each time. On first load, it reads history from the feature store.

For the data source: the RegimeManager needs daily SPX returns and VIX close prices. These come from ORATS historical data (already available) for initial training, and from Tastytrade real-time data for daily updates. The manager accepts raw values — the caller is responsible for providing them.

Add tests for RegimeManager:

- Test update persists to feature store
- Test get_current_regime reads from store
- Test should_retrain triggers on frequent transitions
